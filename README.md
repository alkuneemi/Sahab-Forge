# SAHAB ERP — واجهة React (TypeScript)

إعادة بناء كاملة لصفحات **تسجيل الدخول**، **إنشاء الحساب**، و**لوحة التحكم**
الأصلية (المكتوبة كـ HTML/CSS/JS داخل عقد n8n) كتطبيق React + TypeScript
حديث، مقسّم إلى وحدات (modules) قابلة للتركيب والفك، مع فصل تام بين:

- المنطق/الحالة (`.tsx` / `.ts`)
- التنسيقات (`.css` لكل مكوّن)
- الترجمات (قواميس منفصلة لكل صفحة تحت `src/core/i18n/dictionaries`)

## التشغيل

```bash
npm install
npm run dev      # تشغيل بيئة التطوير على http://localhost:5173
npm run build     # بناء نسخة الإنتاج في dist/
npm run preview   # معاينة نسخة الإنتاج
```

> بيانات المستخدمين والجلسة تُخزَّن حالياً في `localStorage` (لا يوجد
> Backend حقيقي بعد). راجع `src/core/auth/AuthContext.tsx` لاستبدالها
> بطلبات API حقيقية لاحقاً — الواجهة الخارجية (`login`, `signup`, `logout`,
> `session`) لن تتغيّر.

## خريطة المشروع

```
src/
  core/                     ← البنية التحتية المشتركة (لا تُعدَّل عادة)
    auth/                   ← AuthContext (تسجيل الدخول/الخروج + حارس المسارات)
    i18n/                   ← نظام الترجمة (عربي/إنجليزي + RTL/LTR)
      dictionaries/         ← قاموس ترجمة منفصل لكل صفحة
    layout/                 ← SkyShell (الخلفية + الشريط العلوي + الإطار)
    modules/                ← محرّك نظام الوحدات (التركيب/الفك)

  modules/                  ← الوحدات القابلة للتركيب (كل وحدة = مجلد)
    registry.ts             ← السطر الوحيد الذي يُضاف عند كل وحدة جديدة
    sample-theme-module/    ← وحدة مثال جاهزة (دليل الإنشاء بالتعليقات)

  pages/
    auth/                   ← LoginPage / SignupPage + AuthCard المشترك
    dashboard/              ← DashboardPage + كل مكوّناته الفرعية
    system-control/         ← صفحة التحكم بالنظام (تركيب/فك الوحدات)

  shared/components/        ← مكوّنات عامة (حقل نموذج، أزرار OAuth...)
  styles/globals.css        ← متغيّرات التصميم (الألوان، الظلال، الحواف)
  assets/                   ← الصور المستخرجة من الملف الأصلي
```

## نظام الوحدات (Modules) — كيف يعمل الوراثة/التركيب/الفك

كل صفحة وكل تنسيق في المشروع يعتمد على متغيّرات CSS معرّفة مرة واحدة في
`src/styles/globals.css` (مثل `--color-primary`). الوحدة (module) هي حزمة
مستقلة يمكنها:

1. **إعادة تلوين/تنسيق النظام بالكامل** — عبر حقن `<style>` في `<head>`
   يعيد تعريف متغيّرات CSS بعد الملف الأساسي، فتفوز قواعدها تلقائياً
   بسبب ترتيب الإدراج (specificity متساوية + ترتيب لاحق = أولوية).
   عند **فك التركيب** تُحذف وسم `<style>` هذا فتعود التنسيقات الأصلية
   فوراً — بدون أي "تراجع يدوي" أو تعديل في الملفات الأساسية.
2. **إضافة عناصر لقائمة لوحة التحكم الجانبية** (`sidebarItems`).
3. **إضافة صفحات/مسارات كاملة** (`routes`) تظهر فقط أثناء التركيب.
4. **إضافة أو استبدال مفاتيح ترجمة** (`translations`) لكلا اللغتين.

### إنشاء وحدة جديدة (3 خطوات)

1. أنشئ مجلداً: `src/modules/<اسم-الوحدة>/index.tsx`
2. صدّر كائناً يطابق الواجهة `SahabModule`
   (راجع `src/core/modules/types.ts` والمثال الكامل والمُعلَّق في
   `src/modules/sample-theme-module/index.tsx`).
3. أضف الوحدة إلى المصفوفة في `src/modules/registry.ts`.

هذا كل شيء — الوحدة تظهر تلقائياً في صفحة **التحكم بالنظام**
(`/system-control`) بزر "تثبيت"، دون أي تعديل آخر في الكود الأساسي.
عند الضغط على "تثبيت" تُطبَّق تخصيصاتها فوراً على كامل الواجهة، وعند
"إلغاء التثبيت" يعود التصميم الأصلي بالكامل.

## التكامل مع Odoo (الأساس الحقيقي للنظام)

بدل بناء كل تطبيق (مبيعات، مخزون...) من الصفر، الواجهة الآن تعرض بيانات
Odoo **الحقيقية** عبر API، بنفس ثيمنا:

- `src/core/odoo/OdooClient.ts` — عميل JSON-RPC (`{url}/jsonrpc`،
  `common.authenticate`، `object.execute_kw`) — نفس البروتوكول المستخدم في
  سير n8n الأصلي.
- `src/core/odoo/OdooConnectionContext.tsx` — يخزّن بيانات الاتصال ومفتاح
  API لمحلل السجلات الذكي، ويجلب التطبيقات المثبتة (`ir.module.module`).
  كل هذا يُضبط من **صفحة التحكم بالنظام** (`/system-control`).
- **تفعيل تطبيق** من نفس الصفحة يُظهره في قائمة لوحة التحكم الجانبية،
  ويحدَّد لكل تطبيق "Model" أساسي (مع تخمين تلقائي للتطبيقات الشائعة في
  `appModelMap.ts`، وقابل للتعديل يدوياً).
- **عرض التطبيق** (`/apps/:tech`) يجلب `fields_view_get` لعرضي tree و
  form مباشرة من Odoo ويحوّل الـ XML (`viewArch.ts`) إلى شجرة عقد، ثم
  `OdooListView`/`OdooFormView` (في `src/components/odoo-view/`) يرسمانها
  ديناميكياً — **بدون** بناء واجهة يدوية لكل نموذج. يُحترم فيها الإخفاء/
  القراءة فقط/الإلزام (`modifiers`/`attrs` + تقييم `domain`) كما في Odoo
  تماماً.
- بجانب القائمة (75%) يظهر تحليل ذكاء اصطناعي لآخر 50 سجلاً
  (`AiAnalysisPanel.tsx`) يستدعي نقطة النهاية المضبوطة في الإعدادات
  ويعرض HTML يبدأ بالمخاطر ثم التحذيرات ثم الفرص والتوصيات — بنفس تعليمات
  "SAHAB Smart Records Analyst" الأصلية.

> **ملاحظة CORS**: نداءات jsonrpc تحدث من المتصفح مباشرة، لذا يجب أن
> يسمح خادم Odoo بـ CORS من نطاق هذا التطبيق (أو ضع بروكسي بسيط على نفس
> الأصل). هذه أول نقطة تحقّق إن فشل الاتصال.

## إصلاح: `fields_view_get` غير موجودة في Odoo 17+

`OdooClient.getView()` يجرّب الآن `get_views` (الطريقة الحديثة) أولاً،
ويرجع تلقائياً إلى `fields_view_get` فقط إن فشلت — يعمل على أي إصدار Odoo
13 إلى 18 دون أي إعداد إضافي.

## سوق الوحدات الخارجي (مطورون خارجيون بدون كود المصدر)

بالإضافة إلى وحدات `src/modules/` المحلية، يمكن الآن ربط **مستودع GitHub
خارجي** من صفحة الإعدادات — راجع مجلد `sahab-module-sdk/` المرفق بجانب
هذا المشروع: هذا كل ما يحتاجه مطوّر خارجي (لا يحتاج كود SAHAB نفسه). بعد
البناء والرفع إلى GitHub العام، يربط رابط المستودع من الإعدادات فتظهر
وحداته تلقائياً كتطبيقات قابلة للتثبيت/الإلغاء — بنفس آلية الوراثة
(CSS variables + دمج ترجمات + مسارات) المشروحة أعلاه، فقط أن كود الوحدة
يُحمَّل في وقت التشغيل كملف `<script>` مستقل بدل أن يكون جزءاً من هذا
المستودع (`src/core/modules/remote/`).

## حل مشكلة CORS مع Odoo (بدون أي رابط مُثبَّت في الكود)

نداءات `jsonrpc` تمر افتراضياً عبر بروكسي بنفس الأصل (`/odoo-proxy`) بدل
الاتصال المباشر من المتصفح، فلا تصطدم بسياسة CORS الخاصة بأي خادم Odoo —
ودون أي رابط خادم مكتوب داخل الكود؛ كل طلب يحمل رابط Odoo الفعلي (الذي
أدخله المستخدم في الإعدادات) في ترويسة `X-Odoo-Base-Url`.

- **أثناء التطوير**: `vite.config.ts` يحوي بروكسي ديناميكي جاهز — لا شيء
  إضافي مطلوب، يعمل مع أي خادم Odoo تربطه من صفحة الإعدادات.
- **في الإنتاج**: شغّل `npm run proxy` (يستخدم `server/odoo-proxy.mjs`،
  بدون أي مكتبات خارجية) على نفس الخادم، واعمل reverse-proxy لمسار
  `/odoo-proxy` إليه (نفس الأصل)، **أو** انشره على نطاق مستقل وحدّد
  `VITE_ODOO_PROXY_URL` (راجع `.env.example`) قبل `npm run build`.
- إن كان خادم Odoo لديكم يسمح أصلاً بـ CORS من نطاق تطبيقكم، يمكن تجاوز
  البروكسي كلياً بضبط `VITE_ODOO_PROXY_URL=direct`.

### تحديث: سبب خطأ 500 على `/odoo-proxy/jsonrpc`

`server.proxy` المُعلن في Vite مبني على حزمة `http-proxy` التي **لا تدعم
توجيهاً ديناميكياً لكل طلب** (خيار `router` من `http-proxy-middleware` فقط،
وليس من الحزمة التي يستخدمها Vite فعلياً) — فكانت كل الطلبات تذهب فعلياً
إلى الرابط الوهمي الثابت وتفشل بخطأ 500. الحل: استبدلنا ذلك بـ Vite plugin
حقيقي (`configureServer` middleware في `vite.config.ts`) يستخدم نفس معالج
`server/odooProxyHandler.mjs` المُستخدم في الإنتاج — تحكّم كامل بالطلب بدل
الاعتماد على خيارات الإعداد التصريحي المحدودة. كما أصبح `OdooClient.ts`
يتحقق من `res.ok` ويعرض رسالة الخطأ الفعلية بدل فشل تحليل JSON الغامض.

## تحديث: التطبيقات تُفتح بقوائمها الحقيقية من Odoo (وليس بنموذج مُدخل يدوياً)

بدل إدخال اسم "model" يدوياً لكل تطبيق، عند فتح أي تطبيق مفعّل الآن:

1. `OdooClient.getAppMenus(tech)` يجد قائمة الجذر الخاصة بالتطبيق (عبر
   `ir.model.data`) ثم يمشي شجرة `ir.ui.menu` الكاملة بنفس ترتيب `sequence`
   الذي يستخدمه Odoo — تماماً كما تظهر القوائم داخل Odoo نفسه.
2. تُعرض هذه القوائم كشريط علوي، وتُفتح **أول قائمة في الترتيب تلقائياً** —
   كما يحدث بالضبط عند فتح التطبيق من داخل Odoo.
3. عند اختيار أي قائمة، `resolveMenuAction` يقرأ الـ action المرتبط بها
   ليحدد الـ model الفعلي، ثم تُجلب له نفس عروض tree/form الحقيقية كما في
   السابق.

هذا يزيل تماماً أي إعداد يدوي للنماذج من صفحة التحكم بالنظام.

## تحديث: عرض استمارات Odoo الحقيقية (بما فيها العروض الموروثة)

### لماذا لم أُعِد بناء دمج `inherit_children_ids` يدوياً

`OdooClient.getView()` يستدعي `get_views` (أو `fields_view_get` للإصدارات
الأقدم) — وهاتان الطريقتان **تُرجعان من خادم Odoo نفسه الـ arch بعد دمج
كل العروض الموروثة النشطة تلقائياً** (هذا بالضبط ما تفعلانه داخلياً: تطبيق
`position="before/after/inside/replace"` من كل صف في `ir.ui.view` الذي
`inherit_id` يشير لعرضنا). فجلب `arch_base` الخام من `ir.ui.view` وإعادة
تنفيذ محرك xpath هذا يدوياً في المتصفح يُعيد اختراع شيء يقوم به Odoo فعلاً
بشكل أدق وأضمن — لذا أبقيت `get_views` هو المصدر الوحيد للـ arch، وهو
يتضمن العروض الموروثة (مثل الزر/الحقل الذي أرسلتَه كمثال) دون أي كود
إضافي.

### ما الذي أضفته فعلياً لجعل استمارة "Sales Order" الحقيقية تُعرض بلا أخطاء

المشكلة الحقيقية لم تكن الوراثة بل أن **Odoo 17+ يكتب شروط
`invisible`/`readonly`/`required` كتعبير Python مباشر** وليس كـ JSON
(`invisible="state != 'draft'"` بدل `attrs="{'invisible': [...]}"`
القديمة). أضفت:

- **`src/core/odoo/expr.ts`** — مُقيّم صغير وآمن (بدون `eval`) لهذه
  التعبيرات: يدعم `and/or/not`، `==` `!=` `>` `<` `>=` `<=`، `in`/`not in`،
  قوائم `['a','b']`، وأسماء حقول تُقرأ من السجل الحالي. أي تعبير غير
  متوقع يُقيَّم كـ "غير مخفي" بدل أن يُسقط الاستمارة بالكامل.
- **`resolveModifiers`** (في `viewArch.ts`) يجرّب هذا التعبير المباشر
  أولاً، ثم يرجع لصيغة `modifiers`/`attrs` القديمة (Odoo <17) كحل احتياطي.
- **`RecordFormModal.tsx`** أصبح يتحمّل تعقيد استمارة حقيقية كاملة: أزرار
  ملفوفة بـ `<t groups="...">` (يُفكّها الآن قبل البحث عن الأزرار)،
  حقول `one2many`/`many2many` (تُعرض كملخص "N سجل" بدل إخفائها بالكامل)،
  و`<label for="...">` بلا `string` (يأخذ التسمية من بيانات الحقل نفسه)،
  وكل شارات التحذير (`<div class="alert">`) التي تظهر كأشقاء لـ `<sheet>`
  وليست فقط داخله.

**حدود معروفة**: الحقول من نوع `binary` (رفع ملفات) والقوائم المتداخلة
داخل one2many (مثل جدول "Order Lines" التفصيلي بصفوفه القابلة للتعديل)
غير مُنفَّذة بعد — تحتاج محرري جداول متداخلة كاملين، وهي ميزة كبيرة
منفصلة يمكن إضافتها لاحقاً دون التأثير على ما سبق.

## تحديث: حقول العلاقات كاملة + محرر جدول متداخل حقيقي + أخطاء Odoo الفعلية

- **`RelationFields.tsx`**: حقل many2one بحث حي حقيقي (`name_search`) بدل
  إدخال ID يدوي، مع زر "↗" لفتح السجل المرتبط في نافذة جديدة (يجلب عرضه
  وبياناته من Odoo مباشرة). حقل many2many بنفس البحث + وسوم قابلة للحذف.
- **`OneToManyField.tsx`** — محرر جدول متداخل كامل، وليس ملخصاً للقراءة
  فقط: يقرأ السجلات الحالية، يدعم إضافة/تعديل سطر عبر نموذج فرعي حقيقي
  (نفس `RecordFormModal` بشكل تكراري، بوضع "محلي" لا يحفظ في Odoo مباشرة
  بل يُعيد القيم لتُدمَج في حفظ السجل الأب)، وحذف سطر — وعند الحفظ يُنتج
  أوامر الكتابة الصحيحة التي يفهمها Odoo (`[0,0,vals]` إنشاء،
  `[1,id,vals]` تعديل، `[2,id,0]` حذف)، تماماً كآلية One2many الحقيقية.
  يقرأ الاستمارة والقائمة الفرعيتين مباشرة من داخل الحقل نفسه إن كانتا
  مضمّنتين في الـ arch (كما في مثال "Order Lines" الذي أرسلته)، وإلا
  يجلبهما من Odoo عند الحاجة.
- **رسائل الخطأ**: أي فشل في الحفظ أو تنفيذ زر (`write`/`create`/الأزرار
  من نوع object) يُعرض بنص Odoo الفعلي القادم من الخادم (وليس رسالة عامة)،
  بما في ذلك رسائل التأكيد (`confirm="..."`) قبل تنفيذ أزرار مثل "إلغاء"
  أو "إبطال المعاملة".

**الحد المتبقّي**: حقول `binary` (رفع ملفات) ما زالت غير مدعومة —
تحتاج widget رفع/معاينة ملفات منفصل عن نطاق هذا التحديث.

## تحديث أخير: لا شيء معلّق

- **رفع الملفات (binary)**: `BinaryField.tsx` — رفع وتنزيل الملفات بصيغة

- **استيراد CSV**: زر "استيراد" في شريط الأدوات يقرأ ملف CSV، يحوّل القيم

  كتبويب عائم دائم بجانب "عرض أساسي". يعيد استخدام **نفس** نقطة نهاية
  الذكاء الاصطناعي المضبوطة لتحليل السجلات (حقل واحد في الإعدادات يكفي)
